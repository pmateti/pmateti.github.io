This article is a collection of hints to do a decent specs of DITAA tool.

  1. The functional specs should be done in discrete math, logic and grammar.
  2. The functional specs of most non-interactive programs can be done in three parts. This works for DITAA.
    1. The specification of input text. Divide it into a seq of DITAA ASCII diagrams and ordinary text.
    2. The specification of output text. Also, divide it into a seq of DITAA EPS diagrams and ordinary text.
    3. Spec the relationship between an ASCII diagram A and its EPS E. Is this relation a function?
  3. Pages/paragraphs of Specs Doc should reference Pages/paragraphs of Requirements Doc.
  4. [* These hints are written without studying the design/code of DITAA. As you know, there were no specs or requirements written by the author(s) of DITAA. There are several examples. But, I did take a quick look at the submitted Spec Docs. Not thoroughly checked for correctness. *]
  5. For now, the hints below are addressing only the grammar specification.

1 DITAA Terminals (Lexemes)

  1. The examples we considered so far (Telegram Problem, Condense Num Ranges, Tabulate Equations, Common Words Problem, and others) were based on linear text.
  2. DITAA input is based on 2-D text. We should use graph grammars (see refs). But, we have not covered this topic. So, we apply the linear-text grammars, with a few adjustments.
  3. In a linear-text, tokens are separated by delimiters (spaces or special characters). The order of their occurrence matters, but actual x-axis positions do not.
  4. In DITAA, every terminal element has an anchor and its position is given by the line number (y-coordinate), and char position (x-coordinate). For each terminal, we define where its anchor is. We follow the CS cursor position tradition and write it as (y, x). Our coordinate numbers are natural numbers (non-neg integers).
  5. As defined above, the lexical analyzer code will be quite different from that of a typical PL.

2 Box

  1. Box (upper case B) is a non-terminal. Production rules are given below.
  2. tbox is a terminal. The box made of hyphens, vertical bars, +/, the tags inside is this terminal. The contained text, if any, does not become part of this terminal.

2.1 Box/tbox Attributes

  1. Recall that in PLs, (i) "identifier" is a token name, it has the spelling attribute, (ii) "integer" is a token name, it has the numerical value attribute.
  2. A tbox terminal, similarly Box non-terminal, has several attributes. In the following, we group them to save space in this article.
  3. anchor: yx-coord; width, height: nat; corners lt, rt, lb, rb: {r, s}; fill-color: color;
  4. shape modifier tags: doc, storage, io
  5. Dashed lines: {true, false} (true => Colon or = present )
  6. Point markers: seq of yx-coord
  7. Splits: Think of them as juxtapositions horizontally, that is l-to-r, notationally +, or vertically, that is above-below, /.

2.2 Box Production Rule

  1. Box ::= tbox | Box + Box | Box / Box

3 Arrow/Line

  1. Arrow (upper case A) is a non-terminal. Production rules are given below. Line when there is no arrow head.
  2. tail, head: yx-coord. The tail is also its anchor.
  3. Dashed lines: {true, false} (true => Colon or = present )
  4. Turning points: marked with +
  5. Point markers: marked with *

4 Text

  1. Text is the non-terminal. text is the terminal.
  2. For you to work it out.

5 DITAA Non-Terminals

  1. DITAA non-terminals are: Box, Arrow, Line, and Text.
  2. A few examples of the productions are given below in the example.

6 DITAA Input Element DI

  1. The input will be parsed from the given text file into derivations with non-terminals and tokens.
  2. [* Because our "linear text" orientation, we can continue to think of the tokens as l-to-r and above-and-below sequences. *]
  3. [* But all the tokens have anchor points. So, we can even think of the parsed result as a set, instead of a seq, of tokens. *]
  4. The tokens and the derivations are ASCII diagrams.
  5. Anchor positions are integers.
  6. DD ::= Box | Arrow | Line | Text | DD + DD | DD / DD

7 DITAA Output Element DO

  1. The tokens and the derivations are EPS entities.
  2. Anchor positions are the same integers, but scaled.
  3. Each input element is correctly drawn as an EPS entity. [* You can fill in the details. *]
  4. The EPS produced parses exactly the same as the input did.

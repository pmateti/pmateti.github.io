Abstract: This article jots down a few "gotchas" in using assertions in a real Java program. In the src code for every Java class, our goal is to include the class invariant as part of (i) both a pre- and a post-condition for every public method, and (ii) a post-condition of every constructor.

1 SmallSet Implementation Context

  1. ./smallset-impl.html implemented in Java
  2. Includes assertions. Does not use JML.
  3. This is NOT an implementation of Java Set container.
  4. Note: The code may contain bugs, and other issues. It is intended to be used in various teaching purposes.

2 Abstraction Function

  1. setOf(SmallSet s) is the abstraction function. Intended to be used only in assertions.
  2. The code for setOf() uses compact().

2.1 Compact

  1. compact(s) is a function, based on the information hiding principle, removes duplicates, while preserving the abstraction of s. So, the object s before and after abstractly-equals the same setOf(s), as in, taking liberties with our notation, setOf(s.before) == setOf(s.after).
  2. We happen to have declared compact(s) public. So, we must have classInv. setOf(this) == setOf(newset).
  3. You can see the recursion. This will be endless, because depending on s, there is no escape.

3 Code of Assertions

  1. The expressions used in pre- post and class invariants ought to be disjoint from the code of all methods. Had we done the assertions in JML, this would have been clear. Why? Bugs in the methods will alter the true/false results of assertions.
  2. For several best-practice-without-JML reasons, the SmallSet.java implementation did not do the above. It "reused" the code of a couple of public and private methods.

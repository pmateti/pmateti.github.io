2.1 What is Distributed Computing?

  1. Distributed Computing is coordinated computing using machines at many locations.
  2. It primarily uses message passing.
  3. It tries to avoid synchronization.
  4. It tries to avoid shared variables.
  5. It tries to be robust in the presence of individual machine failures.

5.1 Concurrency Fundamentals #1: Non-Determinacy

  1. Ex: Consider the simple assignment statement
    1. {x := 3 [] 7} (Note the fat-bar)
    2. What is the value of x?
    3. Either 3 or 7; nothing else; but which??
    4. Code better be prepared for either.
  2. The above computation P is non-determinant. Each time we run, the result of P may change "for no good reason." Why? Because "non-determinacy" [] is baked in.

5.2 Concurrency Fundamentals #2: Meta Terminology

  1. Fundamentals v Principles v Foundations
  2. Concurrency Fundamentals constitute the core of this course.
  3. Until the last week of this course, we will use concurrency as a term covering all of networked-, parallel-, and distributed- computing.

5.3 Semaphores Defs

  1. A semaphore s is an OOP ADT (abstract data type) variable. It has an integer private data member s, only operable with two public methods called P and V, apart from being initialized.
  2. P(s) ≡ ‹ await s > 0 → s := s-1 ›
  3. V(s) ≡ ‹ s := s+1 ›
  4. The complications in the defs are (i) the atomic angular-brackets and (ii) the meaning of await.

5.4 Starvation-Free Mutual Exclusion

  1. {semaphore m := 1; P(m); Process-i; V(m)}
  2. This is starvation-prone in a distributed context.
  3. Designing starvation-free solutions is part of this course.

5.5 Starvation-Free MutEx: One Solution Idea

  1. The dots are processes. The thick-rectangles are gates letting in one process at a time. The black rectangle is where we require starve-free mutex. A process finishing this will exit as the black dot did, skipping over the black-, red- and green boundaries.
  2. What is the rest of the "idea"?

5.6 Axiomatics #1

  1. Axiomatics is a way of defining the meaning of PL constructs, in a way much better than English prose.
  2. Notationally, we use the triplet < P, S, Q > whre P and Q are assertions (True/False Boolexp) on process state, S is a piece of code.
  3. Q describes the state after S completed. It assumes that P was true when code S started.
  4. P is called a pre-condition. Q is called a post-condition.

5.7 Axiomatics #2

  1. Here is an example applied to semaphore def.
  2. P(s) can be defined as follows: Given [(R and s > 0) ⇒ Q(s, s-1)], we can conclude <R, P(s), Q>
  3. V can be defined as follows: Given [R ⇒ Q(s, s+1), we can conclude <R, V(s), Q>
  4. There will be considerable discussion on this.

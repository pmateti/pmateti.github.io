2 Problem Context: Share a Secret among a Group

  1. Consider a group of 5 people X1, X2, X3, X4, and X5. X1 wants to share a file F1 with the others. [pm: Is the number of Xi the same as number of "shares"?]

2.1 Visual Cryptography (VC)

  1. X1 will apply VC secret sharing scheme (1, 3, 5). The 5 indicates that F1 will be "split" into pieces (called shares in VC) S1, S2, S3, S4, and S5.
  2. The 1 in (1, 3, 5) indicates that one piece among the 5 pieces, say S1, is essential, in the following sense. The 3 indicates that the essential piece and any two other shares are necessary and sufficient to regenerate the file F1.

2.2 IPFS Deployment

  1. All the 5 pieces Si are stored in the IPFS. [pm: Do we store F1?] [pm: Did we apply asymmetric crypto?]
  2. The hash [pm: Five hashes?] generated by IPFS is [are] given to everyone in the group. Each individual of the group will maintain a chain [pm: ??] that stores hash of the share along with the file ID [pm: F1?].
  3. Retrieve on demand basis: When some nodes want to recreate the file the shares need to recombine with the essential share.

2.3 IPFS Deployment #2

  1. Requirement: Others (!= Xi) should not be able to access F1 or the S1.
  2. Requirement: Other pieces (S2, …, S5) can be viewed by anyone. [pm: Decrypted?]

4.3 Q2: Pin Semantics [contd]

  1. direct = pin that specific object.
  2. recursive = pin that specific object, and indirectly pin all its decendants
  3. indirect = pinned indirectly by an ancestor (like a refcount)
  4. From jbenet Mar 7, 2015

4.4 Q3: Peer IDs

  1. Q3: Is the peer ID unique?
  2. A3: [By AmritaU] Yes, Peer ID is unique for the system/ node. Though multiple export action performed finally when we tried to retrieve IPFS ID, the initial value is displayed ( peer id generated at the time of initial installation).
  3. Peer IDs are unique. Across the world. This is similar to https://en.wikipedia.org/wiki/Universally_unique_identifier.

6 Prerequisites
7 =============
8 
9 This course expects that you are already comfortable in the use of
10 networked machines based on Linux/Unix and X11. Learn to login to remote
11 machines via ssh without having to interactively type your password each
12 time; read [2350/ NetworkLab
13 \#ssh](http://cecs.wright.edu/~pmateti/Courses/2350/Labs/Network/NetworkLab.html#ssh)
14 
15 Students whose Linux OS programming skills are rusty should study
16 examples for {`alarm, fork, pipes, signals, bash` scripting} much
17 simpler than what you see in our project files. At WSU, we expect CEG
18 4350/6350 OS Internals and Design to have taught these. Recommended
19 book: W. Richard Stevens and Stephen A. Rago, Advanced Programming in
20 the Unix Environment, Addison-Wesley, ISBN: 0-201-56317-7.
21 <http://www.apuebook.com/>
22 
23 Some of you worry this is C and not C++. C is a subset of C++. Also, the
24 code does use lower-than-what-is-typical-in-C++ libraries. One of the
25 goals of this course: Expose you to the basics of systems programming
26 relevant to distributed computing. If you know C++ well enough, you
27 should be able to fully understand C. Some quick equiv C v C++:
28 `malloc/free` v `new/delete` mem allocation; `static` v `private` scope;
29 `static` v `static` life time; `e1? e2 : e3` v `e1?
30 e2 : e3` conditional expression;
31 
32 RPC background is not expected. Several of the RPC lib procedures will
33 be discussed in class. On Linux, these have become part of C std lib; on
34 other OS you may have to explicitly give `-lrpcsvc` to the linker (`ld`
35 of `gcc` or `g++`).

76 Details
77 -------
78 
79 1. `rpcgen server.x` generates
80  `server.h server_xdr.c server_clnt.c server_svc.c`
81 
82 2. `rpcgen client.x` generates
83  `client.h client_xdr.c client_clnt.c client_svc.c`
84 
85 3. `client_s.c` is a minor variation of `client_svc.c` mechanically
86  produced by running an editor script on `client_svc.c`. We insert
87  `startclient(...)` just above the `svcrun()` line.
88 
89 4. Just the `xwindow.c` can be compiled as a stand-alone program
90  `xwindow7370` to check for X11 problems.
91 
92 5. Note the use of `static int i` within procedures; such an `i` is not
93  allocated on the stack. This makes `i` \"stay/live\" even after that
94  procedure returns.
95 
96 6. Keep the compiler flags `-Wall -ansi -pedantic -std=c99` on; read-up
97  on what these do. There should be zero errors/ warnings. The
98  `rpcgen` generated code unfortunately is not as kosher. Note also
99  that non-Linux OS may have ancient rpcgens that generate even worse
100  code so far as these flags are concerned.
101 
102 7. Our code uses `alarm`, `fork`, `pipe` s, and `signal` s. Our (simple
103  minded) use of signals requires `-D_BSD_SIGNALS.`
104 
105 8. The `% make` will compile the source code file `deregall.c` into a
106  program named `deregall7370`. This can de-register transient RPC
107  processes.
108 
109 9. To clean up the project dir, invoke `% make clean`.
110 10. To archive a version, do `% make tar`

112 How to Run It
113 =============
114 
115 Some of you may wish to use your own Linux boxes for 7370 work. But
116 Linux installations out-of-the-box are typically not \"open\" enough for
117 us. Check your Linux Setup first.
118 
119 1. Host names below can be IP addresses (better/ simpler actually). In
120  the OSIS lab, we use private IP addresses `192.168.17.*`
121 
122 2. In the source code as given: The server is one process running on
123  host H1. The client consists of two processes both running on host
124  H2. It is ok, but not desirable that H1 == H2. H2 is where the X11
125  server is also running, using the bash-equaivalent of `export
126  DISPLAY=H2:0.0`
127 
128 3. Run the \"server\" as a background process on a machine of your
129  choice.\
130  `% server7370 &`
131 
132 4. Run the \"client\" from one or more machines of your choice. The
133  client needs four arguments:\
134  `% client7370 <srv-mchn-nm> <wb-nm> <wb-disp> <color>=\\
135  =<srv-mchn-nm>` is the name of the machine running your server;
136  `<wb-nm>` is the name of the whiteboard you wish to create/join;
137  `<wb-disp>` is the name of the machine you are logged in for the X11
138  display; `<color>` is a 6-hex-digit X11 RGB color number for lines
139  of this client;
140 
141 Here are a few bash procedures: A simple \"smoke test\" with all the
142 processes running on `localhost` and kill all and deregister 7370
143 related processes. Read `man killall` and `man pkill`.
144 
145 ``` {.bash}
146 smokeTest7370() {
147  ./server7370 &
148  echo -n Server started, press Return to continue; read
149  ./client7370 localhost b0 localhost ff0000 & # RGB=ff0000 = red
150  echo -n client started, press Return to continue; read
151  ./client7370 localhost b0 localhost ff00 & # RGB=00ff00 = green
152  echo -n client started; press Return to continue; read
153  ./client7370 localhost b0 localhost ff & # RGB=0000ff = blue
154  echo -n client started, press Return to continue; read
155  rpcinfo -p
156 }
157 
158 killall7370() {
159  killall -q server7370 client7370
160  ./deregall7370 536871065 # change this to your server num
161  ./deregall7370
162  rpcinfo -p
163 }
164 
165 freshStart7370() {
166  killall7370
167  smokeTest7370
168  ps
169  rpcinfo -p
170 }
171 ```
172 
173 Locate where `rpcinfo` is. Perhaps it is in `/usr/sbin` instead of
174 `/usr/bin`. Read its man page. Here is an example invocation on Linux
175 machines:
176 
177 ``` {.bash}
178 % rpcinfo -p
179  program vers proto port
180  100000 2 tcp 111 portmapper
181  100000 2 udp 111 portmapper
182  ... more lines
183 ```
184 
185 Do the above after a fresh reboot. This shows the presence of other
186 standard RPC processes. After you \"close\" your WB project demo, the
187 list then should be the same.
